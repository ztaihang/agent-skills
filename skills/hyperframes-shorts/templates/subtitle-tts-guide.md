# 字幕拆条 + TTS 读音指南（Agent 写 lines.json 前必读）

> **目标 1：** 底部字幕 **永远单行、读得懂** — 既不 stupid  mid-phrase 断开，也不一条塞满被裁切。  
> **目标 2：** TTS **念对** — 多音字、英文/专有名词有 `speak` 兜底。

---

## 一、`lines.json` 字段

| 字段 | 必填 | 用途 |
|------|------|------|
| `id` | ✅ | 如 `s2a`、`s2b` |
| `scene` | ✅ | 对应 HTML `scN` |
| `text` | ✅ | **TTS 原文**（见 §「标点分工」— **保留标点**，含句末。！？，） |
| `speak` | 否 | **仅 TTS 念法**；不写则念 `text`。标点规则同 `text` |
| `notes` | 否 | Agent 备注：为何这样拆、多音字说明（不进 TTS） |

---

## 标点分工（TTS vs 底部字幕 — 常问）

| 环节 | 要不要标点 | 说明 |
|------|-----------|------|
| **`lines.json` → `text`（给 TTS）** | **要，保留** | 逗号、句号、问号、叹号都保留，**含每条末尾的。！？，** — Edge TTS 靠标点控制停顿和语气 |
| **`speak`（可选）** | **同 `text`** | 除非 SSML；专有名词改写时标点照旧 |
| **底部 `.sl` 上屏字幕** | **句末不要** | `apply-audio-schedule.mjs` 自动去掉每条 **末尾** 的 `，。！？：；、` |
| **句中标点** | 字幕可保留 | 如「因此，全球」— 只 strip **末尾**；中间逗号若存在会保留（尽量少在中间加逗号） |

**一句话：**  
- 写 `lines.json` 时按 **正常中文书写带标点** → TTS 自然  
- 观众看到的底部字幕 = **同句文字，但去掉这一条最后一个标点**（你理解得对）

**拆条时标点怎么写：**

```json
[
  { "id": "s3a", "text": "根据最新数据显示，" },
  { "id": "s3b", "text": "2024 年全球 AI 投资规模已经突破 1000 亿美元大关。" }
]
```

- TTS：`s3a` 念完有 **逗号停顿**；`s3b` 念完有 **句号收束**
- 字幕上屏：`根据最新数据显示` / `2024 年全球 AI 投资规模已经突破 1000 亿美元大关`（末尾标点被 strip）

**禁止：** 为了「字幕不要句号」而在 `text` 里删掉句号 — 会导致 TTS 连读发飘。

```json
[
  {
    "id": "s2a",
    "scene": 2,
    "text": "我会自动让我的电脑完成一个任务。",
    "notes": "短句示例：未超上限则一条；断句错误见 §3.3"
  },
  {
    "id": "s5a",
    "scene": 5,
    "text": "OpenClaw 不够聪明？",
    "speak": "Open Claw 不够聪明？",
    "notes": "英文词间加空格，避免 TTS 连读成乱码"
  },
  {
    "id": "s6a",
    "scene": 6,
    "text": "去银行办理业务。",
    "speak": "去银行办理业务。",
    "notes": "行 háng — 若仍读错可改为 speak「去银航办理业务」仅音频（慎用，优先 SSML）"
  }
]
```

## 二、动态单行字数上限（超长句才拆 — 不是「15 字规则」）

**原则：**

1. 用 `meta.json` 算出本项目的 **`maxHan`**（横屏通常约 **16**，竖屏约 **12**；20 字、24 字都可能出现）
2. **`visualUnits ≤ maxHan`** → **一条** `lines.json`，不拆
3. **`visualUnits > maxHan`** → **必须拆**，但只能在 **语义边界** 切（§三），禁止拦腰断
4. 拆完 **每一条** 仍须 `≤ maxHan`；仍超 → 继续找下一断点，**禁止**硬切字数

### 2.1 从项目分辨率算 `maxHan`

读项目 `meta.json` 的 `width` / `height`（无则默认 1920×1080 横屏）：

| 画幅 | 条件 | 建议 `maxHan`（汉字） | 保守 `maxHan`（交付用） |
|------|------|----------------------|-------------------------|
| 横屏 | width ≥ height | `min(18, floor(width×0.88/(字号×1.05)))` | **≤16** |
| 竖屏 | height > width | `min(14, floor(width×0.88/(字号×1.05)))` | **≤12** |

- 字幕字号：横屏 `.sl` 默认 **58px**，竖屏 **52px**（见 SKILL 第十一步）
- **英文/数字**：每个 ASCII 字符按 **0.55 个汉字宽度** 计入总长
- 公式：`visualUnits = 汉字数 + asciiCount×0.55`；`visualUnits ≤ maxHan` 才允许一条

**短句示例（不必拆）：**  
「我会自动让我的电脑完成一个任务」未超横屏上限 → **一条**。

**超长句示例（必须拆）：**  
「如果你不想每天重复同样的操作，你可以让 Agent 自动帮你完成整个工作流。」≈ **28 字** → 超 16，必须拆 2–3 条，见 §3.5。

### 2.2 拆条决策树

```text
1. visualUnits(text) ≤ maxHan？
   → 是：一条 lines.json，结束
2. 有句号/问号/叹号/分号/逗号？
   → 优先在 **最强标点** 处拆（。！？ > ； > ，）
3. 仍超长：按 **语义级** 拆（见 §三），每段再检查 visualUnits ≤ maxHan
4. 拆完后逐条读相邻两句连读 — 若听感像被拦腰斩断 → 合并或改断点
5. 跑 TTS 前：`python scripts/generate-tts.py` 内置校验，ERROR 则改 lines.json
```

## 三、语义拆条规则（禁止 stupid 断句）

### 3.1 允许断在哪里

| 优先级 | 断点类型 | 示例 |
|--------|---------|------|
| P0 | 句号、问号、叹号后 | 两个完整句 → 两条 |
| P1 | 分号、逗号后（子句完整） | 「……，因此……」在逗号后 |
| P2 | 并列连词前 | **但** / **所以** / **因为** / **而且** / **同时** |
| P3 | 动宾之间（新动作起头） | 「我会自动让 \| 电脑完成……」✅（让+宾语不拆） |
| P4 | 话题切换 | 「说完 A。 \| 再来看 B」 |

### 3.2 禁止断在哪里（红线）

| 禁止 | 反例 | 应怎么做 |
|------|------|---------|
| **量词/结构中间** | 「我会自动让 \| 我的电脑……」 | 改断点：「让电脑」不可拆；或上条收在逗号/连词后 |
| **「的」字结构** | 「我的 \| 电脑」 | 禁止；保持「我的电脑」在同一条 |
| **介宾短语** | 「在 \| 桌面上」 | 禁止 |
| **专有名词中间** | 「Open \| Claw」 | 禁止；整条或 speak 加空格 |
| **数字+单位** | 「100 \| 万」 | 禁止 |
| **英文单词中间** | 「Git \| Hub」 | 禁止 |
| **上一句以虚词/介词/助词结尾** | 「……让我的」+「电脑……」 | **最典型错误** — 改断点或合并 |
| **下一句以宾语/名词孤起** | 「电脑完成一个任务」（上句 ending 让我的） | 把「让电脑」放同条或上条带「让」 |

### 3.3 断句错误示例（结构问题，与字数无关）

口播：**我会自动让我的电脑完成一个任务**（即使未超上限，也 **禁止** 下面这种拆法）

| ❌ 错误 | 原因 |
|--------|------|
| `我会自动让我的` + `电脑完成一个任务` | 断在「我的」后，下条以宾语孤起 |
| `我会自动` + `让我的电脑完成一个任务` | 上段过短、听感碎（若未超上限应合并为一条） |

| ✅ 若整句未超 maxHan | 一条写完 |
| ✅ 若与其他内容合并后超长 | `我会自动让电脑` + `完成一个任务`（「让电脑」同条） |

### 3.4 超长句示例（>maxHan，必须拆）

**原文（约 28 字）：**  
「如果你不想每天重复同样的操作，你可以让 Agent 自动帮你完成整个工作流。」

| ❌ 错误 | 原因 |
|--------|------|
| `如果你不想每天重复同样的` + `操作，你可以让…` | 断在「同样的」后 |
| `…帮你完成整个` + `工作流` | 断在专有结构中间 |
| 一条塞满 28 字不拆 | 字幕超出宽度，被裁切 / ellipsis |

| ✅ 推荐 | TTS text | 字幕上屏（自动 strip 末标点） |
|--------|----------|------------------------------|
| 在逗号后拆 | `如果你不想每天重复同样的操作，` | `如果你不想每天重复同样的操作` |
| 第二段 | `你可以让 Agent 自动帮你完成整个工作流。` | `你可以让 Agent 自动帮你完成整个工作流` |

若第二段仍 `> maxHan`，再在 **「Agent」前** 或 **「自动」前** 找 **连词/短语边界**，**禁止**「自动帮你 \| 完成整个」。

**财经长句（约 32 字）：**  
「根据最新数据显示，2024 年全球 AI 投资规模已经突破 1000 亿美元大关。」

```json
[
  { "id": "s3a", "scene": 3, "text": "根据最新数据显示，" },
  { "id": "s3b", "scene": 3, "text": "2024 年全球 AI 投资规模已经突破 1000 亿美元大关。" }
]
```

若 `s3b` 仍超 maxHan → 再拆为「2024 年全球 AI 投资规模，」+「已经突破 1000 亿美元大关。」（**禁止**在「1000 \| 亿」处切）。

### 3.5 Agent 超长句工作流（≥20 字常见）

```text
1. 计算整句 visualUnits
2. ≤ maxHan → 一条（带句末。！？）
3. > maxHan → 列出所有逗号/分号/句号位置
4. 选最靠近中间的 **完整子句** 边界先试拆
5. 对每一段再算 visualUnits，仍超则继续拆
6. 相邻两条连读试听 + generate-tts.py 校验
7. 截帧看 .sl：无 ellipsis、无第二行
```

## 四、TTS 读音（多音字 + 专有名词）

### 4.1 策略顺序

1. **改 `speak` 而不改 `text`**（上屏字幕保持正确写法）
2. 英文/数字：**空格、连字符、中文谐音**（仅 speak）
3. 仍错：换 Edge 音色试读，或 `speak` 用 SSML `<phoneme>`（高级）
4. 在 `notes` 记录，方便用户改稿

### 4.2 常见多音字（口播前逐句扫）

| 字 | 易错场景 | speak 建议 |
|----|---------|-----------|
| 行 | 银行、行业 | 确认读 **háng**；与「行走」同句时注意断句 |
| 重 | 重要 vs 重复 | 重要 zhòng；重复 chóng |
| 还 | 还是 hái / 归还 huán | 按语境选词或 speak 换「仍然/归还」 |
| 得 | 得到 dé / 说得 de / 得去 děi | 按语法 |
| 了 | 了解 liǎo / 完成了 le | 了解 liǎo |
| 地 | 地方 dì / 慢慢地 de | |
| 数 | 数据 shù / 数一数 shǔ | |
| 调 | 调查 diào / 调整 tiáo | |
| 率 | 概率 lǜ / 效率 shuài（口语） | 金融视频常出现，speak 写「绿」或「帅」谐音慎用 |

**Agent 流程：** 写满 `lines.json` 后，**逐条默读**；标红多音字 → 填 `speak`。

### 4.3 专有名词 / 英文 / 产品名

| 类型 | text（字幕） | speak（TTS） |
|------|-------------|-------------|
| 英文产品 | `OpenClaw` | `Open Claw` 或 `Open claw` |
| 缩写 | `API` | `A P I` 或 「接口」 |
| 型号 | `GPT-4o` | `G P T 四 O` 或 「GPT 四 o」 |
| 人名/公司 | 保持原文 | 必要时中文谐音仅 speak |
| 已被 TTS 读坏的词 | 字幕正确汉字 | speak 用 **同音替代词** 或 SSML |

**禁止：** 字幕改成谐音字（「表达 DNA」→ 上屏也写谐音）；**上屏正确，speak 兜底**。

### 4.4 SSML（仅 `speak` 字段，可选）

```json
{
  "text": "去银行办理业务。",
  "speak": "<speak xmlns=\"http://www.w3.org/2001/10/synthesis\" xml:lang=\"zh-CN\">去<phoneme alphabet=\"sapi\" ph=\"hang 2\">行</phoneme>办理业务。</speak>"
}
```

若 Edge 报错，退回 speak 自然语言改写。

## 五、与上屏文案的关系

| 层 | 规则 |
|----|------|
| **口播原文** | 用户稿子，拆镜依据 |
| **lines.json `text`** | = TTS 原文（与口播一致，**保留标点含句末**；可因拆条分段） |
| **上屏大字** | 可精简美化，**含义不变** |
| **底部 `.sl`** | = `text` **仅去掉该条末尾**标点（`，。！？：；、`）；与 TTS 同句；单行 |

**禁止：** 上屏一句、字幕拆成两半且对不上口播节奏。

## 六、交付前字幕/TTS 自检

- [ ] 每条 `visualUnits ≤ maxHan`（或已拆条）
- [ ] **无** §3.2 禁止断点；**超长句**已按 §3.4–3.5 拆且每段 ≤ maxHan
- [ ] **未超 maxHan 的句未无故拆碎**；可合并的相邻短句未过度拆分
- [ ] **`text` 保留 TTS 标点**；确认 `.sl` 仅无 **末尾** 标点（由 apply-audio-schedule 处理）
- [ ] 多音字/专有名词已检查，`speak` 已填或 `notes` 说明
- [ ] `generate-tts.py` 校验 **0 error**（warning 需人工确认）
- [ ] 戴耳机 spot-check：拆条接缝处连读是否自然
- [ ] 截帧：字幕 **无 ellipsis 长期遮挡**（说明仍超长，继续拆或略减字）
